## 一、产品概述

### 1.1 什么是 Multi-Publish

Multi-Publish（社媒管家）是一款面向内容生产者的桌面端工具，提供 **"采集 → 改写 → 发布"** 全流程闭环的一键发布体验。

> ![产品主界面概览](images/main-interface.png)
> *图 1：Multi-Publish 主界面 — 左侧导航栏 + 右侧内容区*

**核心工作流：**

```
┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│  内容创作  │ → │ 内容采集  │ → │ 内容编辑  │ → │ 一键发布  │
│ (外部工具) │   │ URL/粘贴  │   │ 模板/优化  │   │ 多平台分发 │
└──────────┘    └──────────┘    └──────────┘    └──────────┘
                                                    │
                                              ┌─────┴─────┐
                                              │ 任务队列管理 │
                                              │ 实时进度追踪 │
                                              └───────────┘
```

### 1.2 核心价值

| 能力 | 说明 |
|------|------|
| **统一入口** | 一个桌面应用管理 15 个平台的发布 |
| **自动适配** | RPA 自动化填表发布，适配各平台 UI |
| **异步队列** | 后台批量发布，实时追踪状态 |
| **多账号管理** | 同平台可添加多个账号，一键切换 |
| **定时发布** | 设定时间自动发布，关机后重启恢复 |
| **安全存储** | Cookie AES-256-GCM 加密存储 |
| **离线可用** | 安装包自带 Chromium，无需额外下载 |

### 2.2 下载安装

**方式一：GitHub Releases（推荐）**

1. 访问 [GitHub Releases](https://github.com/Colinchiu007/Multi-Publish/releases)
2. 根据系统下载对应安装包：

| 系统 | 安装包 | 说明 |
|------|--------|------|
| Windows | `Multi-Publish.Setup.x.x.x.exe` | 双击安装 |
| macOS | `Multi-Publish-x.x.x.dmg` | 拖入 Applications |
| Linux | `Multi-Publish-x.x.x.AppImage` | `chmod +x` 后运行 |

**方式二：Gitee 国内镜像**

国内用户推荐从 [Gitee Releases](https://gitee.com/ColinChiu007/Multi-Publish/releases) 下载，330MB 文件通常 1-2 分钟完成。

**方式三：源码构建**

```bash
git clone https://github.com/Colinchiu007/Multi-Publish.git
cd Multi-Publish
npm install
npm run dev    # 开发模式
npm run build:win  # 构建 Windows 安装包
```

### 4.2 多账号管理

**同平台多账号：**

Multi-Publish 支持同一平台添加多个账号，适合运营多个账号的团队。

> ![多账号切换](images/multi-account-switch.png)
> *图 11：侧栏多账号切换 — 点击账号头像快速切换*

**操作方式：**

| 操作 | 方法 |
|------|------|
| 添加多个账号 | 在同一平台多次点击"+ 新增账号" |
| 切换账号 | 侧栏点击平台下的账号头像 |
| 设置默认账号 | 账号管理页面点击星标 ★ |

**状态指示灯：**

| 指示灯 | 含义 |
|:------:|------|
| 🟢 绿色 | Cookie 有效，登录态正常 |
| 🟡 黄色 | Cookie 即将过期（7天内） |
| 🔴 红色 | Cookie 已过期，需重新登录 |

### 4.3 登录状态管理

**自动检测机制：**

- App 启动时自动后台验证所有账号的 Cookie 有效性
- 使用隐藏 BrowserWindow 静默验证（loginSilent）
- 定期轮询检测登录状态

**过期处理：**

1. 账号过期时，状态指示灯变红
2. 发布时会自动检测并提示重新登录
3. 点击 **「重新登录」** 按钮 → 重新走登录流程
4. 新 Cookie 自动加密存储

### 4.4 三种认证模式

| 模式 | 实现方式 | 适用场景 |
|------|---------|---------|
| **内嵌浏览器登录** | WebContentsView 内嵌登录页面 | 大多数平台的标准登录 |
| **隐藏浏览器静默验证** | 隐藏 BrowserWindow 恢复 Cookie 检测 | 自动续期、后台检测 |
| **扫码登录** | QrCodeLogin 自动检测二维码（img/canvas 3 策略） | 微信生态平台 |

### 4.5 安全机制

| 安全措施 | 说明 |
|---------|------|
| **Cookie 加密** | AES-256-GCM 加密算法，密钥本地派生 |
| **凭证存储** | credential-store.js 统一管理，不落盘明文 |
| **Session 隔离** | `session.fromPartition()` 每个账号独立 Cookie 分区 |
| **不上传服务器** | 所有凭证仅存储在本地，不会上传到任何远程服务器 |

### 27.2 API Key 管理

**创建 Key：**

1. 设置 → **「API 管理」**
2. 点击 **「创建 API Key」**
3. 设置权限范围（scope）：
   - `*` 通配符：全部权限
   - `publish:read`：只读发布任务
   - `publish:write`：提交发布任务
   - `admin`：管理权限（创建/吊销 Key）
4. 生成的 Key 格式：`mp_` + 48 位十六进制字符
5. **复制并保存 Key**（关闭后不再显示原文）

**管理 Key：**

| 操作 | 端点 | 说明 |
|------|------|------|
| 创建 | `POST /api/v1/keys` | 新建 Key（需 admin scope） |
| 列表 | `GET /api/v1/keys` | 查看所有 Key（不暴露原文） |
| 吊销 | `DELETE /api/v1/keys/:key` | 吊销指定 Key |

### 27.3 API 端点

| 方法 | 端点 | 说明 |
|:----:|------|------|
| GET | `/api/v1/platforms` | 获取支持平台列表 |
| POST | `/api/v1/publish` | 提交发布任务 |
| GET | `/api/v1/jobs/:id` | 查询任务状态 |
| POST | `/api/v1/keys` | 创建 API Key（admin） |

| DELETE | `/api/v1/keys/:key` | 吊销 API Key（admin） |

const status = await client.getJobStatus(job.id);

## 三十、常见问题

### 21.1 安装与启动

**Q: 安装包下载很慢怎么办？**

A: 推荐以下方式：
- 使用 **Gitee 国内镜像**：https://gitee.com/ColinChiu007/Multi-Publish/releases（5MB/s）
- 使用 `gh-proxy.com` 加速：`https://gh-proxy.com/https://github.com/Colinchiu007/Multi-Publish/releases/download/v1.1.0/Multi-Publish.Setup.1.1.0.exe`

**Q: 首次启动很慢，卡在"检测依赖"？**

A: 首次启动需要下载内置浏览器组件（~170MB），请耐心等待 1-2 分钟。如果超过 5 分钟无响应：
1. 检查网络连接
2. 检查是否被防火墙拦截
3. 重启 App 重试
4. 首次运行后，后续启动秒开

**Q: macOS 提示"无法验证开发者"？**

A: 由于未进行 Apple 开发者签名，首次打开需要：
1. 右键点击 App → 选择「打开」
2. 在弹出的对话框中选择「打开」
3. 之后即可正常打开

**Q: Linux 下 AppImage 无法运行？**

A: 执行 `chmod +x Multi-Publish-*.AppImage` 赋予执行权限后运行。如果提示 FUSE 错误，添加 `--no-sandbox` 参数。

### 21.2 账号相关

**Q: 添加账号时浏览器窗口不弹出？**

A: 检查：
1. 是否被防火墙拦截
2. 操作系统弹窗权限是否开启
3. 是否有其他 Electron 应用占用端口

**Q: 登录状态经常过期怎么办？**

A: 各平台登录有效期不同（微信扫码通常 7-30 天，账号密码可能 1-3 天）。建议：
- 定期检查账号状态（状态指示灯 🟢=正常 🟡=即将过期 🔴=已过期）
- 收到重新登录提示时及时处理
- 微信公众号使用扫码登录有效期更长

**Q: Cookie 安全吗？会被上传到服务器吗？**

A: 绝对安全：
- 所有 Cookie 使用 **AES-256-GCM** 加密后存储在本地文件
- **不会上传到任何远程服务器**
- 每个账号使用 `session.fromPartition()` 独立 Cookie 分区
- 加密密钥从本地机器特征派生

**Q: 最多能添加多少个账号？**

A: 没有数量限制。但建议单平台不超过 10 个账号，过多的账号同时发布可能被平台识别。

### 21.3 发布相关

**Q: 发布失败怎么办？**

A: 按以下步骤排查：

| 步骤 | 检查项 | 解决方法 |
|:----:|--------|---------|
| ① | 账号登录状态 | 重新登录过期账号 |
| ② | 查看失败原因 | 任务详情中显示具体错误信息 |
| ③ | 内容格式 | 检查是否有不兼容的格式 |
| ④ | 敏感词 | 运行敏感词检测 |
| ⑤ | 重试 | 修复后点击「重新发布」 |

**常见失败原因：**

| 原因 | 处理方式 |
|------|---------|
| Cookie 过期 | 重新登录 |
| 内容违规（敏感词） | 修改内容后重试 |
| 平台限流 | 等待 5 分钟后重试 |
| 网络异常 | 检查网络连接 |
| 平台 UI 变更 | 更新到最新版本 |

**Q: 视频上传支持哪些格式和大小？**

A: 支持 `mp4`、`mov`、`avi` 格式，建议不超过 500MB。各平台对视频时长和分辨率有不同限制，发布前请参考平台官方要求。

**Q: 定时任务重启后还在吗？**

A: 在。定时任务持久化到 SQLite 数据库，App 重启后自动恢复。

**Q: 为什么提示"发布间隔太短"？**

A: 同一账号两次发布之间至少需要间隔 **5 分钟**，这是为了防止被平台限流封号。等待 5 分钟后自动重试。

**Q: 批量发布时，一个平台失败会影响其他平台吗？**

A: **不会**。每个平台的发布任务互相独立，一个平台失败不影响其他平台的发布。

### 21.4 数据与存储

**Q: 数据存储在哪里？**

A: 所有数据存储在本地应用数据目录：

| 系统 | 默认路径 |
|:----:|---------|
| Windows | `%APPDATA%/Multi-Publish/` |
| macOS | `~/Library/Application Support/Multi-Publish/` |
| Linux | `~/.config/Multi-Publish/` |

**Q: 如何备份数据？**

A: 备份应用数据目录下的 `database.sqlite` 文件和 `config/` 目录即可。

**Q: 更换电脑如何迁移数据？**

A: 将旧电脑上的应用数据目录整个复制到新电脑的对应位置，覆盖后重启 App。

### 21.5 其他

**Q: 能否同时在多台电脑使用？**

A: 免费版限 1 台，Pro 版限 3 台。

**Q: 如何联系技术支持？**

A: 通过以下渠道：
- **GitHub Issues**：https://github.com/Colinchiu007/Multi-Publish/issues
- **Pro 用户**：享有优先级支持（48 小时内回复）

**Q: 如何提交功能建议？**

A: 在 GitHub Issues 提交 Feature Request，或通过设置页的「反馈」入口提交。

**Q: 是否会收集用户数据？**

A: 仅收集使用统计数据（如发布次数、功能使用率），用于改善产品体验。**不会收集**内容数据、账号密码、Cookie 等敏感信息。
