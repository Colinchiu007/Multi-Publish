# 全面项目测试与需求验证

## 触发条件
当需要对项目进行全面的前端界面和功能操作验证时使用。

## 执行要求

### 1. 范围界定
- **界面验证**：逐项对照 PRD 中的所有 UI 需求，验证每个界面元素的内容、状态、交互是否正确
- **功能验证**：完整走通 PRD 中定义的所有功能操作流程，每个流程至少执行两遍，验证功能正确性和稳定性
- **数据一致性**：操作过程中验证数据在前端、状态、存储之间的一致性

### 2. PRD 对照
- 逐条梳理 PRD 中的功能点，形成检查清单
- 每完成一项测试，立即在清单上标记结果
- 发现 PRD 有但实现无，或实现与 PRD 不符的项，记录为差异

### 3. 操作流程
- 主流程（Happy Path）：正常操作流程走通
- 异常流程（Edge Cases）：边界条件、错误处理、空状态
- 两遍验证：同一流程执行两遍，确保结果一致

### 4. 测试方法
应用质量节拍，按以下顺序执行：
- Phase 2（开发期）：走日常循环，验证核心功能
- Phase 3（交付期）：/qa 完整质量检查 + /review 代码审查
- Phase 4（复盘期）：汇总测试结果，沉淀 learnings

### 5. 输出格式

**测试报告包含：**
- 测试时间、范围、环境
- 通过项 / 失败项 / 差异项清单
- 每个失败项的：步骤、操作、预期结果、实际结果、截图（如有）
- PRD 差异对照表
- 修复建议

### 6. 质量门禁
- 核心流程全部通过才能标记为"测试完成"
- 任何 P0/P1 级问题未解决不得合入
