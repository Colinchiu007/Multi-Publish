## 七、关键问题（Growth Areas）

### ❌ 1. 复盘文档严重膨胀（41+ 轮）

**证据**: 7-11 当天生成 50+ 轮复盘文档，每次小修复都写 retro。

**根因**: 把 "复盘" 当成 "提交日志"，失去沉淀价值。

**改进**:
- 复盘应该按 "主题" 而非 "轮次" 组织
- 每周 1 次主题复盘，而非每 commit 1 次
- 删除 41 轮历史复盘，归档为 1 份 "7 月审查期经验总结"

### ❌ 2. 38 轮代码审查反映缺陷预防不足

**证据**: 7-10 当天 30+ 次 "fix: 第N轮审查修复" 提交。

**根因**: 缺陷在开发阶段未拦截，堆积到审查阶段批量修复。

**改进**:
- 开发前先用 /plan-eng-review 锁定架构
- 开发中用 6 步日常循环的 "测试脑暴" 步骤
- 审查改为抽样而非全量

### ❌ 3. 工作时间无边界（24 小时提交）

**证据**: 凌晨 00-06 时有 15+ 次提交。

**根因**: 单人项目无作息约束 + AI 协作降低执行门槛。

**改进**:
- 设定硬性下线时间（22:00 后不提交）
- 用 /context-save 保存进度，次日 /context-restore 恢复

### ❌ 4. 测试滞后于修复

**证据**: fix(66) >> test(17)，比例 3.9:1。

**根因**: 先修复后补测试的惯性思维。

**改进**:
- TDD 强制：fix 提交必须包含回归测试
- 用 /qa 在 fix 后立即验证

### ❌ 5. 文档反复重写

**证据**: PRD.md 单次 885 行重写 + mojibake 乱码修复。

**根因**: 一次性大改 + 编码处理不当。

**改进**:
- 文档小步迭代，单次不超过 200 行
- 统一 UTF-8 无 BOM 编码

## 八、可复用模式（已沉淀到 project_memory.md）

### Pattern 1: 大组件拆分三步法

```
1. 提取 composables（保持主文件 API 不变）
2. 收缩主文件（替换 inline 逻辑为 composable 调用）
3. 补 composable 单元测试（独立可测）
```

### Pattern 2: Python 模块拆分

```
1. 识别 500+ 行的 God Module
2. 提取独立职责为 xxx_brief/xxx_compiler 模块
3. 原模块 import 提取的模块，保持外部 API 不变
4. 补新模块独立测试
```

### Pattern 3: Bridge Pattern (Python Sidecar)

```
spawn Python subprocess + HTTP 调用 + watchdog auto-restart
适用于：Electron/Node.js 集成 Python 服务
```

### Pattern 4: 质量节拍卡点

```
Phase 0-5 全流程 + 6 步日常循环
适用于：单人 + AI 协作的项目质量保障
```
