## 三、策略分歧深度分析

### 3.1 分歧点 #1：`bootstrap.js` 是否应该拆分？

| 维度 | 方案 A | 方案 B |
|------|--------|--------|
| **立场** | 必须拆分（P0） | **暂不拆分**，延后到 TS 落地后 |
| **理由** | 429 行承担 5+ 种职责，高风险单点 | 作为 DI 入口，拆分涉及事件注册顺序和 IPC 时序，风险更高 |
| **建议拆分为** | di-setup / task-queue-wiring / event-bus-setup / platform-config / index | — |
| **方案 B 的补充理由** | — | 待 TS 类型检查落地后，编译器可帮助发现时序错误 |

#### ✅ 我的修正判断：

**方案 B 更稳妥。** 理由：
1. bootstrap.js 的复杂性主要来自「接线」而非「逻辑」，拆分后文件间依赖仍然紧密
2. 事件注册顺序是 Electron 应用的隐性契约，拆分容易破坏
3. **折中方案**：不拆分文件，但内部提取子函数并添加分段注释，将认知复杂度从 O(n) 降到 O(1)

```javascript
// 折中的 bootstrap.js 内部重组（保持单文件）
function createAppContext() {
  const container = createContainer()
  
  // ══════════════════════════════════════
  // PHASE 1: 基础设施初始化
  // ══════════════════════════════════════
  initInfrastructure(container)
  
  // ══════════════════════════════════════
  // PHASE 2: 任务系统
  // ══════════════════════════════════════
  const taskSystem = initTaskSystem(container)
  
  // ... 后续阶段
}
```

### 3.2 分歧点 #2：TypeScript 迁移策略

| 维度 | 方案 A | 方案 B |
|------|--------|--------|
| **策略** | 「全面迁移到 .ts」或「统一 JSDoc+checkJs」（二选一） | **明确选 JSDoc+checkJs**，反对重命名 .js→.ts |
| **理由** | TS 提供完整类型安全 | 重命名会**破坏 Node.js CommonJS require 链**（Node 默认只解析 .js/.json/.node） |
| **批次划分** | 未细化 | 3 批次：core(2文件) → 基础工具(10文件) → 业务模块(11文件) |

#### ✅ 我的修正判断：

**方案 B 正确。** 这是一个关键的技术决策：

```javascript
// ❌ 方案 A 原始想法（危险）
// 重命名 container.js → container.ts
// 结果：require('./container') 在运行时会找 container.js，找不到！

// ✅ 方案 B 的正确做法（安全）
// 保持 container.js，添加 JSDoc 类型注解
/** @type {import('./types').Container} */
const container = new Container()
```

**采纳方案 B 的 TS 迁移策略，吸收其 3 批次划分。**

- `require('./logger')` 被 **25+ 文件**引用
- `container.setup.js` 有 **30+ 处**服务注册

**修正后的建议**：采用方案 B 的 barrel file 策略，降级为 C5 可选任务。

### 3.4 分歧点 #4：实施粒度和验证门禁

| 维度 | 方案 A | 方案 B |
|------|--------|--------|
| **阶段划分** | 5 个 Phase（每阶段 1-3 周） | **3 个阶段**（A:1-2h / B:7-9h / C:10-15h） |
| **时间单位** | 周 | **小时**（更精确） |
| **打包验证** | 模糊提到 | **QM-1 不可跳过**（每个子任务后 electron-builder --win） |
| **回滚机制** | 未定义 | **明确的回滚协议**（30min 无法修复则 revert） |
| **手动冒烟** | 未要求 | **每阶段必须**（启动→登录→发布→确认无崩溃） |
| **提交策略** | 未定义 | **git 小步提交**，每个子任务一个 commit |

#### ✅ 我的修正判断：

**方案 B 的工程实践明显更成熟。** 我的学习点：

1. **时间估算应以小时为单位**，周级估算容易拖延
2. **Electron 应用必须打包验证**，仅跑测试不够（测试不覆盖原生模块集成）
3. **手动冒烟测试不可替代**，Electron 的时序问题自动化测试难以捕获
4. **回滚协议必须预先定义**，否则重构卡住时进退两难

## 五、关键启发总结

### 从方案 B 学到的 5 件事

1. **数据优于直觉**：方案 B 用 `npm audit` / `depcheck` / 测试运行数据说话，避免了我在评分时的主观偏差
2. **量化风险比描述风险更有价值**：「100+ 处 require 变更」比「影响面大」 actionable 得多
3. **TypeScript 迁移不能想当然**：`.js` → `.ts` 重命名在 Node.js CommonJS 下会静默破坏应用
4. **Electron 重构必须有打包验证门禁**：测试通过了不代表打包后能正常运行
5. **审查自己的方案能发现盲区**：方案 B 的 25 条修正项说明第一版方案总有遗漏

### 从方案 A 学到的 5 件事（方案 B 可补充的）

1. **平台配置分散是一个独立的架构问题**，不应被淹没在「巨型模块拆分」里
2. **Scheduler 重复实现的合并有具体的性能和维护收益**
3. **DI 容器的工厂函数 bug 是一个真实的运行时缺陷**（可能导致意外的 singleton 行为）
4. **前端架构也是重构范围的一部分**（方案 B 完全忽略了渲染进程）
5. **量化 ROI 能帮助争取资源和排优先级**
