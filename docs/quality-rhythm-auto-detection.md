# 质量节拍自动检测逻辑

## 目的
确保所有涉及代码修改的任务都强制触发质量节拍，不遗漏任何场景。

## 自动检测规则

### 1. 文件修改检测
在执行以下操作前，必须自动触发质量节拍：
- apply_patch - 应用代码补丁
- git add - 添加文件到暂存区
- git commit - 提交代码
- write_file - 写入文件
- delete_file - 删除文件
- rename_file - 重命名文件

### 2. 用户意图检测
用户消息包含以下关键词时，必须自动触发质量节拍：
- 实现、修复、重构、优化、添加、修改、编辑、创建、删除
- 功能、模块、组件、页面、视图、接口、API
- Bug、错误、异常、问题、故障
- 测试、验证、检查、审查

### 3. 任务类型检测
当前任务属于以下类型时，必须自动触发质量节拍：
- feature - 新功能开发
- bugfix - Bug修复
- refactor - 代码重构
- optimization - 性能优化
- config - 配置变更
- documentation - 文档更新

### 4. 文件类型检测
即将修改的文件属于以下类型时，必须自动触发质量节拍：
- JavaScript/TypeScript文件：.js, .ts, .jsx, .tsx
- Vue文件：.vue
- Python文件：.py
- 其他编程语言文件：.java, .go, .rs, .c, .cpp, .h
- 配置文件：.json, .yaml, .yml, .toml
- 样式文件：.css, .scss, .less

## 检测流程

用户请求/即将执行操作
        |
        v
+-----------------+
|  自动检测引擎   |
+-----------------+
        |
        |-- 1. 文件修改检测 --> 检测到文件操作 --> 触发质量节拍
        |
        |-- 2. 用户意图检测 --> 检测到意图关键词 --> 触发质量节拍
        |
        |-- 3. 任务类型检测 --> 检测到代码任务类型 --> 触发质量节拍
        |
        +-- 4. 文件类型检测 --> 检测到代码文件 --> 触发质量节拍
                |
                v
        +-----------------+
        |  执行质量节拍   |
        +-----------------+
                |
                |-- 检查变更类型
                |-- 评估变更规模
                |-- 路由到对应Phase
                +-- 输出检查清单

## 代码修改前置检查

在执行任何代码修改操作前，AI必须自动检查：

1. 是否即将执行文件修改操作（apply_patch, git add, git commit等）
2. 用户消息是否包含代码相关关键词
3. 当前任务是否涉及代码实现、修复、优化等
4. 即将修改的文件是否是代码文件

检测到任一条件 -> 立即触发质量节拍，不等待用户确认。

触发方式：执行 $quality-rhythm 或使用触发词

### 方式2: Wrapper函数
使用 scripts/quality-rhythm-wrapper.js 包装所有代码修改操作：

const qualityRhythmWrapper = require('./scripts/quality-rhythm-wrapper');

// 包装apply_patch
const wrappedApplyPatch = qualityRhythmWrapper.wrapApplyPatch(originalApplyPatch);

// 包装git操作
const wrappedGitAdd = qualityRhythmWrapper.wrapGitOperation(originalGitAdd);

### 方式3: Pre-commit Hook
在 .git/hooks/pre-commit 中添加质量节拍检查：

#!/bin/bash
echo "质量节拍检查中..."
node scripts/quality-rhythm-wrapper.js check
if [ $? -ne 0 ]; then
    echo "质量节拍检查失败，禁止提交"
    exit 1
fi

## 违反处理

### 未触发质量节拍
- 后果：不允许提交代码
- 处理：Code Review打回，要求补充质量节拍检查

### 跳过质量节拍流程
- 后果：视为流程违规
- 处理：记录违规行为，纳入质量评分

### 绕过强制检查
- 后果：视为严重违规
- 处理：暂停开发权限，进行安全审计

## 监控和统计

### 触发率统计
- 统计所有代码修改任务中触发质量节拍的比例
- 目标：100%触发率

### 违规统计
- 统计未触发质量节拍的代码修改任务
- 目标：0%违规率

### 质量评分
- 根据触发率和违规率计算质量评分
- 目标：质量评分 >= 90分
