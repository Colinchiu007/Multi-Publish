# PROJECT-003：多平台一键发布 — PRD

> **立项日期**: 2026-06-03  
> **产品定位**: 为内容生产者提供"采集 → 改写 → 发布"全流程闭环的一键发布工具  
> **目标用户**: 自媒体运营者、企业内容团队、SEO 运营

---

## 一、产品概述

### 1.1 核心价值

内容生产者每天需要在多个平台发布相同或相似的内容。手动操作耗时、易出错、格式不统一。PROJECT-003 提供：

1. **统一入口**：一个界面管理所有平台的发布
2. **自动适配**：自动调整内容格式适配各平台规范
3. **异步队列**：后台批量发布，实时追踪状态
4. **Cookie 管理**：安全存储各平台登录凭证

### 1.2 产品边界

| 范围 | 说明 |
|------|------|
| ✅ 包含 | 微信公众号（官方 API）、知乎（RPA，P2）、微博（预留）、抖音（预留） |
| ⏳ 待定 | 小红书（风控极高，暂不纳入） |
| ❌ 不包含 | 掘金、CSDN（已移除）、内容创作（由 PROJECT-001 负责）、视频发布（由 PROJECT-002 负责） |

| 平台 | 优先级 | 技术路线 | 难度 | 工期 | 理由 |
|------|--------|----------|------|------|------|
| **微信公众号** | P0 | 官方 API | 2/5 | 2-3天 | 有官方 API，最稳定，商业价值最高 |

## 三、功能需求

### 3.1 核心功能

#### F1：平台账号管理

| 子功能 | 描述 |
|--------|------|
| 添加平台 | 选择平台类型，配置登录凭证（API 或 Cookie） |
| 凭证加密 | 所有敏感信息加密存储（AES-256） |
| 状态检测 | 定期检测账号是否有效（Cookie 过期/API 权限） |
| 一键切换 | 支持同一平台多个账号（如多个公众号） |

#### F2：内容发布

| 子功能 | 描述 |
|--------|------|
| 单篇发布 | 手动输入标题 + 内容 → 选择平台 → 发布 |
| 批量发布 | 选择多篇内容 → 选择多个平台 → 批量发布 |
| 定时发布 | 设置发布时间 → 后台定时任务执行 |
| 草稿模式 | 发布为草稿（不直接公开） |

#### F3：自动格式适配

| 子功能 | 描述 |
|--------|------|
| 标题适配 | 各平台标题长度限制（公众号 64 字、知乎 50 字等） |
| 正文适配 | Markdown → 各平台格式（HTML/富文本） |
| 图片适配 | 自动上传封面图、调整尺寸 |
| 标签适配 | 各平台标签/分类体系映射 |

#### F4：任务管理

| 子功能 | 描述 |
|--------|------|
| 任务队列 | 后台异步执行，支持优先级 |
| 实时进度 | WebSocket 推送发布进度 |
| 结果通知 | 成功/失败通知，失败原因记录 |
| 重试机制 | 失败自动重试（可配置次数） |

### 3.2 非功能需求

| 需求 | 指标 |
|------|------|
| 并发发布 | 支持同时发布到 5 个平台 |
| 响应时间 | 单篇发布 < 30s |
| 数据加密 | 所有凭证 AES-256 加密存储 |
| 日志审计 | 所有发布操作留痕，可追溯 |

### 4.3 发布器接口规范

每个发布器必须实现以下接口：

```python
from abc import ABC, abstractmethod
from dataclasses import dataclass
from typing import Any

@dataclass
class PublishResult:
    success: bool
    platform: str
    article_id: str | None = None
    url: str | None = None
    error: str | None = None

class BasePublisher(ABC):
    @abstractmethod
    async def publish(self, title: str, content: str, **kwargs) -> PublishResult:
        """发布内容"""
        pass
    
    @abstractmethod
    async def check_auth(self) -> bool:
        """检查认证状态"""
        pass
    
    @abstractmethod
    async def close(self):
        """关闭资源（如浏览器实例）"""
        pass
```

## 六、与 PROJECT-001 的集成

### 6.1 集成方式

```
PROJECT-001（内容聚合改写）
    │
    │ 改写后的内容
    ▼
┌──────────────────┐
│  一键发布按钮     │ ← 调用 PROJECT-003 API
└────────┬─────────┘
         │
         ▼
PROJECT-003（多平台发布）
```

### 6.2 集成点

1. **API 调用**：PROJECT-001 通过 HTTP 调用 PROJECT-003 的 `/api/publish` 端点
2. **共享配置**：凭证加密库共用（或独立存储）
3. **统一 UI**：未来可整合为单一前端

## 七、风险与应对

| 风险 | 影响 | 应对 |
|------|------|------|
| 微信公众号 API 权限申请 | 高 | 需要企业认证，个人号无法使用 API → 需企业配合申请 |
| RPA 被平台封禁 | 中 | 行为模拟 + 随机延迟 + Cookie 轮换 |
| 平台规则变更 | 中 | 模块化设计，单个发布器变更不影响整体 |
| Cookie 过期频繁 | 低 | 自动检测 + 一键重新登录 |
